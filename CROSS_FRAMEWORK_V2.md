# 美团 Cross 分析竞赛项目最终框架文档 V2

## 1. 项目定位

本项目定位为一个面向商业分析竞赛的完整 Cross 决策系统，而不是单一模型实验。

核心目标不是只回答“用户会不会从 A 跳到 B”，而是系统性回答以下四个问题：

1. 哪些业务/品类/场景之间存在明显的 Cross 机会。
2. 这些 Cross 机会的协同强度、引流效率、转化潜力和价值代理有多高。
3. 哪些 Cross 场景值得优先做，应该在什么时机、针对什么人群、以什么方式去做。
4. 如何把上述分析结果以可交互、可展示、可答辩的方式呈现出来。

建议项目总标题：

`CrossMind：基于异构图学习与智能体决策的美团跨业务协同机会发现与价值评估系统`

---

## 2. 赛题目标与项目交付映射

| 赛题目标 | 项目承接模块 | 最终交付形式 |
|---|---|---|
| 构建业务 Cross 潜力量化评估工具，预估交叉引流效果与转化价值 | 候选发现层 + 价值评估层 | CrossScore 排行榜、价值评估卡、Top 场景清单 |
| 量化不同业务 Cross 的协同效应与引流效率，挖掘高潜力 Cross 场景 | EDA 统计量化层 | 热力图、Lift 矩阵、漏斗图、时间滞后分布 |
| 结合实际业务，提出可落地的业务 Cross 运营策略建议 | 策略引擎层 | HIGH/MEDIUM/LOW 分层建议、业务动作建议表 |
| 总结适用于不同业务组合的分析框架与指标体系 | 总框架文档 + 指标体系层 | 方法论图、指标定义表、实验流程图 |
| 强化展示力与技术力 | Agent 交互展示层 | Gradio/Streamlit Demo、Agent 策略报告 |

结论：Agent 是展示与调用入口，不是核心分析层；核心分析能力仍由 EDA、建模、价值评估、策略生成四部分构成。

## 3. 总体系统架构

本项目采用四层架构：

1. EDA 统计量化层
2. GNN 候选发现层
3. 价值评估与策略层
4. Agent 交互展示层

整体数据流如下：

```text
原始行为日志 view_data.csv
-> 数据清洗与 session 重建
-> EDA 与指标体系
-> 统计/序列/图模型生成候选 Cross 对
-> CrossScore 统一评分与价值评估
-> StrategyTier 运营分层
-> Agent 调用结果生成业务策略报告
-> 报告 / PPT / Demo / 附录
```

各层职责如下：

| 层级 | 核心职责 | 输入 | 输出 | 在最终作品中的意义 |
|---|---|---|---|---|
| EDA 统计量化层 | 发现规律、定义指标、形成业务洞察 | 清洗后的行为数据 | Lift 矩阵、CCR、用户分群、时序窗口 | 解释力基础，决定报告说服力 |
| GNN 候选发现层 | 学习结构关系，发现高潜 Cross 场景 | 异构图数据 | POI/品类级候选 Cross 对 | 技术力主干，体现算法深度 |
| 价值评估与策略层 | 统一打分、排序、给出运营优先级 | EDA 信号 + GNN 结果 + 时序信号 | CrossScore、StrategyTier、建议卡 | 直接回答“值不值得做、怎么做” |
| Agent 交互展示层 | 产品化表达、交互式解释和演示 | 前三层输出 | 策略问答、案例展示、自动报告 | 竞赛答辩亮点，强化完成度 |

### 4.2 能做的事情

- 基于 session 链路分析用户跨场景迁移
- 识别高潜力业务/品类组合
- 评估短链路 Cross 潜力、转化倾向、时机窗口和价值代理

### 4.4 正确表述方式

最终报告和答辩建议统一表述为：

“本项目评估的是短周期行为链路中的 Cross 潜力、转化倾向与价值代理分数，并通过结构化建模为跨业务协同提供数据驱动依据。”

## 5. 研究主线与前后步骤因果关系

### 5.1 主线逻辑

本项目不是“先训练一个模型再找故事”，而是严格按照以下因果链展开：

1. 因为题目要求量化 Cross 协同和引流效率，所以必须先重建用户行为链路。
2. 因为要让结果具备解释性，所以必须先建立 EDA 和指标体系，再进入算法层。
3. 因为业务组合数量巨大，所以必须先做候选发现，再做价值评估，避免在全空间盲打分。
4. 因为“会跳转”不等于“值得做”，所以必须把潜力识别和价值评估拆成两个阶段。
5. 因为竞赛不只看模型分数，还看业务落地，所以在价值评估后必须加入策略层。
6. 因为你希望体现产品化和就业背书，所以最后再用 Agent 封装结果，而不是让 Agent 代替分析。

### 5.2 最终流程

```text
Step 1 数据清洗
-> Step 2 session 重建
-> Step 3 EDA 与指标体系
-> Step 4 候选 Cross 发现
-> Step 5 GNN 学习结构关系
-> Step 6 CrossScore 统一评分
-> Step 7 StrategyTier 生成运营策略
-> Step 8 Agent 调用与展示
-> Step 9 报告 / PPT / Demo 输出
```

## 6. 各阶段详细方案

## 6.1 阶段一：数据清洗与 Session 重建

### 目标

将原始事件日志转化为可用于 Cross 分析的标准行为链路数据。

### 核心动作

- 统一 `event_type` 与 `event_id`，识别 `PV / MC / ORDER`
- 去除关键字段缺失记录
- 按 `session_id + event_timestamp` 对行为排序
- 构造 `next_poi`、`next_category`
- 生成 `time_bucket`、`city`、`device_type`
- 对 POI、用户、品类做全局编码映射

### 输出

- 标准化明细表
- Session 路径表
- 节点映射表
- 后续图建模所需的边表

### 角色与意义

这一步是整个系统的地基。后续 EDA、GNN、策略、Agent 都依赖它。若这一步不稳定，最终结果会失真。

---

## 6.2 阶段二：EDA 统计量化层

### 目标

先用统计和可视化回答“Cross 机会在哪里”，建立解释框架和指标体系。

### 核心分析内容

- 一级品类之间的 Cross 转移热力图
- 业务/品类链路漏斗：`PV -> MC -> next_cate -> ORDER`
- 城市、设备、时段维度的切片对比
- 高频路径与典型消费链路分析
- 用户行为分群：高活跃、单场景深耕、多场景迁移、低活跃等
- 时序滞后分析：访问 A 到访问或下单 B 的时间差分布

### 核心指标体系

1. `CCR`：Cross Conversion Rate  
   在访问 A 的用户中，后续在 B 发生目标行为的比例。

2. `Lift`  
   `P(B|A)` 相对 `P(B)` 的提升，用于剔除热门品类天然高频带来的干扰。

3. `Cross Efficiency`  
   单位曝光或点击带来的跨场景转化效率。

4. `Coverage`  
   对应 Cross 组合可覆盖的人群规模。

5. `Stability`  
   不同城市、设备、时段下该 Cross 组合是否稳定。

6. `Time Window`  
   某 Cross 组合最适合触达的时间窗口。

### 输出

- Cross 热力图
- Lift 矩阵
- 时间滞后分布图
- 用户行为分群图
- 关键业务洞察卡

### 角色与意义

这一层承担双重任务：

1. 作为竞赛直接交付成果，增强解释力。
2. 作为后续算法层的先验知识输入。

---

## 6.3 阶段三：候选 Cross 发现层

### 目标

在全量业务组合中，先筛出值得重点评估的候选 Cross 对。

### 方法组合

1. 统计基线
- 共现频次
- 条件概率
- Lift
- PMI

2. 序列基线
- session 内一步转移概率
- Markov 风格路径分析

3. 图候选
- 基于 `poi -> next_poi`
- 聚合得到 `category -> category`

### 输出

- 候选 POI 对
- 候选品类对
- 高潜业务组合初筛榜单

### 角色与意义

这一层相当于“搜索空间收缩器”，减少无效打分，提高后续模型聚焦能力。

---

## 6.4 阶段四：GNN 候选发现层

### 目标

通过异构图建模学习用户、商家、品类之间的高阶结构关系，发现传统统计方法难以识别的潜在 Cross 场景。

### 异构图设计

#### 节点

- `user`
- `poi`
- `category`

可扩展节点：

- `city`
- `device_type`
- `time_bucket`

#### 边

- `user - pv -> poi`
- `user - mc -> poi`
- `user - order -> poi`
- `poi - belongs_to -> category`
- `poi - transition_to -> poi`

### 任务设计

建议采用双任务思路：

1. 任务 A：预测 `next_poi / next_category`
2. 任务 B：预测某条 Cross 链路后续是否更容易产生转化代理行为

### 推荐模型定位

- 正式主模型：异构图 GNN
- 备用/对照：统计基线、序列基线

### 输出

- POI embedding
- Category embedding
- 候选 Cross 潜力分数
- Top-K 跨场景推荐对

### 角色与意义

这一层是项目最核心的技术证明层，负责展示模型相比简单统计方法的增益。

---

## 6.5 阶段五：价值评估与统一评分层

### 目标

将“会发生跳转”进一步升级为“是否具备商业价值”。

### 统一评分公式

建议采用统一评分表达式：

```text
CrossScore(A, B) = α * S1 + β * S2 + γ * S3 + δ * S4
```

其中：

- `S1`：统计协同强度  
  例如 Lift、CCR、条件转化率、样本量平滑项。

- `S2`：图结构潜力  
  例如 GNN 输出的嵌入相似度、链路预测分数、结构邻近性。

- `S3`：时机成熟度  
  例如时间滞后分布、集中度、最佳触达窗口匹配程度。

- `S4`：稳定性与可覆盖性  
  例如覆盖用户数、城市设备时段稳定性。

### 权重确定方式

权重不应拍脑袋设定，建议通过以下方式选择：

- 验证集调参
- 排序效果对比
- 消融实验

### 二阶段价值评估建议

将价值评估拆成两步更稳：

1. `P(jump to B | A)`：从 A 到 B 的迁移倾向
2. `P(order at B | jump from A)`：迁移后转化倾向

若无金额字段，则最终输出：

`Value Proxy = P(jump) * P(order|jump) * Coverage * Stability`

### 输出

- CrossScore 排行榜
- Top 高价值 Cross 组合
- 不同业务组合的价值评分卡

### 角色与意义

这一层是最贴近赛题主命题的部分，直接回答“Cross 机会值不值得做”。

---

## 6.6 阶段六：策略引擎层

### 目标

将模型与统计结果自动翻译成可执行的业务动作建议。

### 策略分层

建议将结果分为：

- `HIGH`
- `MEDIUM`
- `LOW`

即：

```text
StrategyTier(A, B) = f(CrossScore(A, B))
```

### 分层建议模板

- HIGH：高潜高转化，优先联投、资源位联动、联合活动、时机触达
- MEDIUM：有潜力但承接链路不足，适合定向实验、落地页优化、优惠联动
- LOW：暂不优先，维持观察

### 输出

- 业务对运营建议卡
- 不同城市/设备/时段下的触达建议
- 对应的人群建议与时机建议

### 角色与意义

这一步让项目从“预测模型”升级为“运营决策系统”，也是商业分析竞赛里很容易拉开差距的一层。

---

## 6.7 阶段七：Agent 交互展示层

### 目标

把前三层的能力整合为一个可交互的业务分析助手。

### Agent 推荐工具接口

1. `query_cross_potential(bu_name, top_k=5)`  
   输入业务或品类，返回潜力最高的 Cross 对及其分数、原因和策略层级。

2. `get_cross_user_segment(bu_a, bu_b)`  
   返回最相关的人群行为分群特征，而不是夸大为完整用户画像。

3. `get_time_window(bu_a, bu_b)`  
   返回最优触达时间窗口与时间滞后分析结论。

4. `generate_strategy_report(bu_a, bu_b)`  
   组合上述信息，输出完整策略建议。

### Demo 形式

- Gradio
- Streamlit

### 交互示例

- “外卖首页适合导向什么业务？”
- “饮品到电影的 Cross 潜力为什么高？”
- “广州 iPhone 用户夜间最适合什么 Cross 策略？”

### 输出

- 自然语言策略报告
- 指标卡片
- Top-K 场景推荐
- 证据链解释

### 角色与意义

这一层是答辩场景里的“记忆点制造器”，让评委看到这不是静态报告，而是一个可使用的分析系统。

## 7. CrossScore 在最终结果中的角色

CrossScore 是本项目最关键的“统一表达中枢”，它的价值在于：

1. 将统计规律、结构潜力、时机信息和稳定性整合为一个统一的可排序分数。
2. 让算法结果可以自然连接到策略层，而不是停留在抽象 embedding。
3. 让最终报告有清晰的方法论抓手，便于答辩表达和简历包装。

建议在最终报告里把它包装为：

“本项目提出的 Cross 综合潜力评分框架”

## 9. 实验与评估设计

### 9.1 Baseline

必须保留基线对比，否则很难证明 GNN 的价值。

建议至少包含：

- 共现统计
- Lift 排序
- Markov 一步转移
- GNN 主模型

### 9.2 评估维度

- AUC / Recall@K / HitRate@K
- 排序质量
- 高潜场景命中质量
- 业务解释合理性
- 稳定性分析

### 9.3 消融实验

建议至少做以下消融：

- 只用统计信号
- 只用 GNN 结构信号
- 只用时序信号
- 统计 + GNN
- 统计 + GNN + 时序

### 9.4 案例研究

至少选 2 到 3 个业务场景做案例拆解，例如：

- 外卖 -> 酒旅
- 饮品 -> 电影
- 餐饮 -> 打车

每个案例都给出：

- CrossScore
- 最佳时间窗口
- 目标人群行为特征
- 运营建议

## 16. 最终结论

本项目最优落地方式不是“单模型方案”，也不是“单纯 Agent 包装”，而是一个四层联动的 Cross 决策系统：

```text
EDA 统计量化层
-> GNN 候选发现层
-> 价值评估与策略层
-> Agent 交互展示层
```

现有四个脚本并非无效，而是分别对应了调试训练、正式训练、演示推理、稳定推理模板等不同角色。后续应以 [`cross_analysis_fullgraph_training_v2.py`](/home/konglingrui/meituan_project/cross_analysis_fullgraph_training_v2.py) 为主干，以 [`inference_sample.py`](/home/konglingrui/meituan_project/inference_sample.py) 为推理基础，对整体工程继续模块化重构。

最终目标是把该项目做成一个既能拿比赛成绩、又能体现技术深度、还适合作为简历与面试背书的完整作品。
