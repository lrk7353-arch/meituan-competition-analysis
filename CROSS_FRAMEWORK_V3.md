# 美团 Cross 项目最终框架文档 V3

## 1. 项目定位

本项目 V3 的定位，不再是“单一模型实验”或“Cross 指标分析”，而是一个面向本地生活平台导流决策的完整离线评估系统。

核心目标不是只回答“用户会不会从 A 跳到 B”，而是系统性回答以下五个问题：

1. 哪些业务/品类/场景之间存在可识别的 Cross 候选机会。
2. 这些 Cross 候选的导流价值、协同强度和成交代理有多高。
3. 哪些 Cross 虽然统计上很强，但会破坏用户当前任务主线。
4. 哪些 Cross 值得平台主动推动，适合在哪个流量面、什么时机、以什么方式去做。
5. 如何把结果表达成既能答辩、又能接近平台决策语言的框架化输出。

建议项目总标题升级为：

`CrossMind V3：面向本地生活平台任务连续性的 Cross 导流决策评估系统`

---

## 2. 赛题目标与项目交付映射

| 赛题目标 | V3 承接模块 | 最终交付形式 |
|---|---|---|
| 构建业务 Cross 潜力量化评估工具，预估交叉引流效果与转化价值 | 候选发现层 + 价值评估层 | CrossValueScore 排行榜、Top Cross 清单 |
| 量化不同业务 Cross 的协同效应与引流效率，挖掘高潜场景 | EDA 统计量化层 | Lift 矩阵、CCR 结果、Markov 路径热力图 |
| 结合实际业务，提出可落地策略建议 | 合理性约束层 + 策略层 | TaskFitScore、PushPriority、策略卡 |
| 总结适用于不同业务组合的分析框架与指标体系 | 总框架文档 + 指标定义层 | 方法论图、评分定义、执行流程 |
| 强化展示力与产品化表达 | 决策输出层 + Demo/Agent | 四象限看板、业务问答、自动报告 |

结论：

- Agent 是展示入口，不是核心分析器
- GNN 不再是技术主线
- `Markov + Lift + CCR + 合理性约束` 才是 V3 的正式主流程

## 3. V3 总体系统架构

V3 采用六层结构：

1. 数据底座层
2. EDA 统计量化层
3. 候选发现层
4. 价值评估层
5. 合理性约束层
6. 决策与策略输出层

整体数据流如下：

```text
原始行为日志 view_data.csv
-> 数据清洗与 session 重建
-> 严格订单前序归因
-> baseline v2 生成 Markov / Lift / CCR
-> 候选 Cross 池
-> CrossValueScore
-> TaskFitScore
-> PushPriority / Quadrant / Tier
-> 策略建议 / 报告 / Demo / Agent
```

各层职责如下：

| 层级 | 核心职责 | 输入 | 输出 | 在最终作品中的意义 |
|---|---|---|---|---|
| 数据底座层 | 统一口径、修正订单归因、重建链路 | 原始日志 | 标准行为表、归因表 | 保证全流程可信 |
| EDA 统计量化层 | 建立解释框架和业务洞察 | 标准行为数据 | 热力图、漏斗、时序图、切片结果 | 决定报告说服力 |
| 候选发现层 | 找出高潜 Cross 对 | baseline v2 资产 | 候选 pair 池 | 缩小搜索空间 |
| 价值评估层 | 识别哪些 Cross 值得重点看 | Markov / Lift / CCR | CrossValueScore | 回答“值不值钱” |
| 合理性约束层 | 判断是否符合任务主线 | 时段、链路、语义、终局质量 | TaskFitScore | 回答“该不该推” |
| 决策与策略输出层 | 合成平台动作建议 | CrossValueScore + TaskFitScore | PushPriority、四象限、策略卡 | 回答“怎么推” |

## 4. 数据前提与边界

### 4.1 数据特点

- 数据源：[`view_data.csv`](/home/konglingrui/meituan_project/view_data.csv)
- 主分析底座：[`view_data.v1.1.csv`](/home/konglingrui/meituan_project/view_data.v1.1.csv)
- 时间范围：2026 年 1 月 11 日单日数据
- 数据规模：约 680 万条行为记录
- 主要字段：`user_id`、`session_id`、`event_timestamp`、`first_cate_name`、`page_city_name`、`event_type`

### 4.2 能做的事情

- 基于 session 行为链路分析用户跨场景迁移
- 识别高潜业务/品类组合
- 估计短周期导流价值与成交代理
- 对不同流量面给出策略建议
- 评估不同 Cross 的任务连续性风险

### 4.3 不能过度承诺的事情

- 不能把单日行为数据包装成长期因果增量结论
- 不能宣称获得真实 GMV 增量
- 不能把离线相关性直接说成线上策略必然有效
- 不能把“会发生”误说成“应该推动”

### 4.4 正确表述方式

V3 建议统一表述为：

> 本项目评估的是短周期行为链路中的 Cross 候选价值与任务连续性，并基于两者合成平台导流优先级，为跨业务协同提供离线决策依据。

## 5. 研究主线与步骤因果关系

V3 不是“先训练模型再找故事”，而是严格按照以下因果链展开：

1. 因为要研究 Cross 导流，所以必须先重建可靠的 session 与订单归因链路。
2. 因为要让结果具备解释力，所以必须先建立 `Markov / Lift / CCR` 这三条基线。
3. 因为业务组合很多，所以必须先做候选发现，再做价值评估。
4. 因为“高价值”不等于“值得主动推”，所以必须加入合理性约束层。
5. 因为平台动作要落地，所以不能只给总分，必须给流量面和策略建议。
6. 因为竞赛看完整性，所以最后再接报告、PPT 和 Demo，而不是让展示层替代分析层。

---

## 6. V3 的正式主线流程

```text
Step 1 数据清洗与归因修复
-> Step 2 baseline v2 生成三条主信号
-> Step 3 候选 Cross 池构建
-> Step 4 CrossValueScore 价值评估
-> Step 5 TaskFitScore 合理性约束
-> Step 6 PushPriority 决策合成
-> Step 7 Strategy 输出
-> Step 8 报告 / PPT / Demo
```

注意：

- GNN 不再出现在正式主线中
- 若提及 GNN，只能放在“历史探索 / 弃用路线 / 附录复盘”

## 7. 各阶段详细方案

## 7.1 阶段一：数据底座层

### 目标

把原始行为日志转成可支撑 Cross 决策评估的标准链路数据。

### 核心动作

- 统一 `PV / MC / ORDER`
- 清洗关键字段缺失
- 按 `session_id + event_timestamp` 排序
- 重建 session 内链路
- 基于最近前序显式行为对订单做严格归因
- 标准化业务线与类目字段

### 当前资产

- [`DATASET_USER_ORDER_REPROFILE.py`](/home/konglingrui/meituan_project/DATASET_USER_ORDER_REPROFILE.py)
- [`view_data.v1.1.csv`](/home/konglingrui/meituan_project/view_data.v1.1.csv)

### 输出

- 归因增强明细表
- 可直接供 baseline 使用的统一底座

---

## 7.2 阶段二：EDA 统计量化层

### 目标

在进入决策层之前，先回答“Cross 机会大致在哪里”。

### 关键内容

- 一级业务线热力图
- 类目层典型链路
- 城市 / 设备 / 时段切片
- 高转化链路与强协同链路对照
- 时间滞后分布

### 关键意义

EDA 在 V3 中不只是背景材料，而是：

- 给价值层提供解释锚点
- 给合理性层提供业务先验

---

## 7.3 阶段三：候选发现层

### 目标

从全空间里先筛出值得进入决策评估的 Cross pair。

### 方法来源

- `Markov`：找顺路跳转
- `Lift`：找独特协同
- `CCR`：找成交承接

### 候选池构建建议

候选池不应只取单一榜单，而应做并集：

```text
CandidatePool
= TopK_by_Markov
+ TopK_by_Lift
+ TopK_by_CCR
+ Business mandatory pairs
```

### 作用

这一层负责“缩空间”，不是“做最终裁决”。

---

## 7.4 阶段四：价值评估层

### 目标

衡量一条 Cross 作为业务机会本身值不值得重点关注。

### 当前资产

- [`CROSS_SCORE_V1.py`](/home/konglingrui/meituan_project/CROSS_SCORE_V1.py)

### 当前判断

`CrossScore V1` 是一个有价值的过渡版，但本质上仍偏“价值排序器”。

它已经能做的事情：

- 融合 `Lift / Markov / CCR`
- 加入覆盖和时机因素
- 初步输出 `HIGH / MEDIUM / LOW`

它还没做好的事情：

- 没区分“价值”与“合理性”
- category 层容易出现大盘承接黑洞
- 还没有正式形成平台动作决策逻辑

### V3 升级方向

价值层应正式命名为：

```text
CrossValueScore
```

它回答的是：

> 这条 Cross 作为业务机会值不值钱。

---

## 7.5 阶段五：合理性约束层

### 目标

判断一条高价值 Cross 是否仍然符合本地生活平台的任务完成逻辑。

### 这是 V3 的核心新增层

本地生活平台和内容平台的根本差别在于：

- 内容平台允许发散探索
- 本地生活平台必须对任务完成负责

因此，V3 新增：

```text
TaskFitScore / Rationality Constraints
```

### 建议纳入的信号

1. `主任务保持率`
   A 跳到 B 后，用户是否仍沿原任务链成交。

2. `多跳衰减合理性`
   用于压制“局部合理、全链路失真”的跳转。

3. `场景语义距离`
   用人工 taxonomy 即可先做。

4. `当前时段匹配`
   午餐、夜宵、出行、酒店等时段/场景规则应显式进入评估。

5. `会话终局质量`
   若跳转后高退出、无单、游离，应惩罚。

### 它回答的问题

> 这条 Cross 作为平台动作是否合适。

---

## 7.6 阶段六：决策合成层

### 目标

把“值钱”与“合理”合成平台真正可执行的优先级。

### 推荐表达

V3 不建议再只输出一个总榜，而应输出：

1. `CrossValueScore`
2. `TaskFitScore`
3. `PushPriority`
4. `Quadrant`

### 四象限定义

| 象限 | 含义 | 建议动作 |
|---|---|---|
| Q1 | 高价值 + 高连续性 | 主动推动、重点资源 |
| Q2 | 高价值 + 低连续性 | 小流量验证、加强约束 |
| Q3 | 低价值 + 高连续性 | 补充流量位、轻触达 |
| Q4 | 低价值 + 低连续性 | 过滤或观察 |

### 最终判断逻辑

V3 的最终目标不再是“找出最高分 pair”，而是：

> 找出最值得平台主动推动的高质量 Cross。

---

## 7.7 阶段七：策略输出层

### 目标

把分析结果翻译成平台动作。

### 推荐策略维度

- 首页承接
- 支付后推荐
- 会场联动
- 搜索联动
- 内容种草
- 召回提醒

### V3 的关键变化

策略层不再只根据一个总分输出，而是根据：

- 价值大小
- 合理性高低
- 场景类型
- 流量面匹配度

来决定打法。

---

## 8. 业务线层与类目层的角色分工

### 8.1 业务线层

业务线层是 V3 的正式主结论层。

用途：

- 报告主图
- 答辩主叙事
- 平台级策略讨论

### 8.2 类目层

类目层是 V3 的案例和精细化补充层。

用途：

- 展示代表性链路
- 做玩法案例
- 解释策略差异

原则：

- 不直接把类目 Top 榜当项目主榜
- 必须经过合理性过滤和案例筛选

---

## 9. GNN 在 V3 中的处理方式

V3 必须明确写清楚：

> GNN 当前是弃用状态，不进入正式主流程。

建议在最终作品中这样处理：

- 正文不再以 GNN 为方法主角
- 若需要展示技术探索，可放在附录
- 统一表述为：
  - 历史探索路线
  - 单日薄图条件下效果不稳
  - 因而被正式主线替代

这样会比“硬保留一个已失效辅助层”更诚实，也更有判断力。
