# 美团 Cross 项目技术路线 2.0

## 1. 最终方法选型结论

本项目最终采用的核心技术路线为：

```text
主线方法：Lift / CCR + Markov
辅助方法：GNN
统一输出：CrossScore + 策略建议 + Agent / Demo
```

这意味着项目不再以 GNN 作为唯一主角，而是采用“统计主线 + 序列主线 + 图模型辅助”的分层方案。

最终定位如下：

- `Lift / CCR`：负责发现显著协同关系，提供最强解释力。
- `Markov`：负责刻画单日 session 内的短期迁移规律。
- `GNN`：负责提供结构补充信号，发现长尾和弱显式共现关系。
- `CrossScore`：统一融合多种信号，形成排序和策略层输出。

---

## 2. 为什么从“GNN主线”调整为“统计+序列主线”

这个调整不是因为 GNN 无效，而是因为当前数据形态决定了它不适合单独承担主线角色。

### 2.1 当前数据的关键特征

- 数据时间范围只有一天
- 任务场景是单日行为链路
- 用户 session 较短
- POI 粒度非常细
- 图结构天然稀疏
- 当日偶然共现和局部噪声较多

### 2.2 这会导致什么问题

在单日薄图数据上，GNN 很容易出现以下现象：

- 训练集学习效果看起来很强
- 验证集泛化提升有限
- 容易放大偶然共现
- 容易把局部拓扑当成稳定协同关系
- 为了降低 loss，逐渐出现过拟合倾向

因此，继续把 GNN 当作“唯一答案生成器”，并不符合当前数据条件。

### 2.3 更合理的角色划分

对于这类数据，更适合的做法是：

- 用统计方法抓显著规律
- 用序列方法抓短期迁移
- 用 GNN 补充结构泛化能力

这比“单模型梭哈”更稳，也更适合商业分析竞赛。

---

## 3. 技术路线 2.0 的整体结构

项目采用“三方法 + 一融合层”的结构：

```text
EDA 统计层
-> Lift / CCR

序列迁移层
-> Markov

结构辅助层
-> GNN

融合层
-> CrossScore

输出层
-> 策略建议 / Agent / Demo
```

在职责上：

| 模块 | 角色 | 优势 | 是否主线 |
|---|---|---|---|
| Lift / CCR | 显著协同发现 | 强解释、稳、适合单日数据 | 是 |
| Markov | 短期迁移刻画 | 最贴近 session 路径 | 是 |
| GNN | 结构补充与长尾发现 | 有创新性，可发现弱显式关系 | 辅助 |
| CrossScore | 统一融合 | 把方法结果转成决策输出 | 是 |

## 4. 各方法的职责边界

## 4.1 Lift / CCR：统计主线

### 作用

回答“哪些业务/品类之间确实存在强协同关系”。

### 适合当前数据的原因

- 单日数据中，直接的行为提升关系最可信
- 统计量对短周期数据更稳
- 结果最容易解释给评委和业务方听

### 建议核心指标

1. `Lift(A, B)`
2. `CCR(A, B)`
3. `Coverage(A, B)`
4. `Time Window(A, B)`

### 主要输出

- 一级品类 Lift 矩阵
- 一级品类 CCR 矩阵
- Top Cross 场景榜单
- 时间窗口分布图

### 在最终系统中的定位

作为 `CrossScore` 的核心组成部分，承担 `S1`。

---

## 4.2 Markov：序列主线

### 作用

回答“在单日 session 链路中，用户从 A 之后最可能去哪里”。

### 为什么适合当前项目

- 你的数据本质上就是行为序列
- 单日数据更像短句，不像长期图谱
- Markov 比图神经网络更贴近这种短序列迁移问题

### 方法定义

基于 session 排序构建：

```text
P(B | A)
```

即：

- 给定用户在当前一步访问/点击/下单了 A
- 下一步跳转到 B 的概率

### 主要输出

- 类目转移矩阵
- 业务转移矩阵
- Top-K 下游场景候选

### 在最终系统中的定位

作为 `CrossScore` 的核心组成部分，承担 `S2_seq` 或并入 `S2`。

---

## 4.3 GNN：结构辅助层

### 作用

GNN 不再负责“直接决定最终答案”，而是负责：

- 提供结构相似性补充信号
- 发现统计上不明显但拓扑上接近的长尾关系
- 作为项目创新点和技术亮点

### 当前合理定位

> GNN 是辅助发现器，不是唯一决策器。

### 为什么还要保留

- 能提升作品技术含金量
- 能补充统计与序列方法的盲区
- 可以作为 `CrossScore` 的结构项
- 可以作为答辩中“创新方法”的组成部分

### 在最终系统中的定位

作为 `CrossScore` 的辅助项，承担 `S3_graph`。

## 5. CrossScore 的新定义

在技术路线 2.0 中，CrossScore 不再单纯强调 GNN，而是改成更贴合数据条件的融合方式：

```text
CrossScore(A, B) = α * S_stat + β * S_seq + γ * S_graph + δ * S_time + ε * S_cov
```

其中：

- `S_stat`：统计协同强度  
  例如 Lift、CCR、样本量平滑。

- `S_seq`：序列迁移强度  
  例如 Markov 转移概率。

- `S_graph`：结构潜力  
  例如 GNN embedding 相似度、链路打分。

- `S_time`：时机成熟度  
  例如从 A 到 B 的时间滞后和最佳触达窗口。

- `S_cov`：覆盖度与稳定性  
  例如涉及用户数、城市设备切片稳定性。

### 角色理解

- `S_stat` 决定“是否显著”
- `S_seq` 决定“是否顺路”
- `S_graph` 决定“是否有潜在长尾关系”
- `S_time` 决定“何时触达”
- `S_cov` 决定“值不值得大规模做”

---

## 6. 为什么这种组合比“单一方法”更合理

### 6.1 不是方法堆砌，而是职责分层

这个组合不是为了“看起来很复杂”，而是因为每个模块承担不同任务：

- Lift / CCR：发现显著规律
- Markov：发现路径迁移
- GNN：补充结构长尾

如果只用一个方法，会有明显短板：

#### 只用 Lift / CCR

- 优点：稳、解释强
- 缺点：只能看显式共现，长尾关系容易遗漏

#### 只用 Markov

- 优点：非常贴近 session
- 缺点：只看一步转移，结构泛化不足

#### 只用 GNN

- 优点：技术感强
- 缺点：对单日薄图不够稳，解释弱，过拟合风险高

### 6.2 当前最优组合思路

因此最适合你的最终方法不是“三选一”，而是：

> 统计方法做主锚点，序列方法做主预测器，GNN 做辅助增强器。

## 9. 推荐的开发顺序

基于当前情况，最合理的推进顺序如下：

### 第一阶段：先把统计层做出来

优先完成：

- Lift
- CCR
- 时间窗口

原因：

- 最快产生高价值解释结果
- 也是 `CrossScore` 的基础
- 对答辩最有帮助

### 第二阶段：补 Markov

完成：

- 类目转移矩阵
- 业务转移矩阵
- Top-K 转移候选

原因：

- 更贴合当前单日 session 数据
- 往往比 GNN 更稳

### 第三阶段：保留 GNN 作为辅助

完成：

- embedding 导出
- category 聚合
- 结构辅助打分

原因：

- 形成技术创新亮点
- 作为 `S_graph`

### 第四阶段：统一做 CrossScore

完成：

- 多信号融合
- 策略层分级
- Top Cross 榜单

### 第五阶段：Agent / Demo

完成：

- 业务问答式展示
- 策略建议自动生成

这样听起来就不是“堆方法”，而是“系统设计”。

## 11. 最终技术路线的项目价值

这条路线的优势在于：

1. 贴合当前数据物理形态  
   单日薄图更适合统计和序列主线。

2. 保留技术含金量  
   GNN 仍然存在，但角色更合理。

3. 强解释力  
   Lift / CCR / Markov 都很容易讲清楚。

4. 强可落地性  
   CrossScore 和策略层天然适合接 Agent / Demo。

5. 更符合竞赛需求  
   不只是“跑一个模型”，而是完成“洞察 + 预测 + 策略 + 展示”的闭环。

## 12. 最终结论

本项目的技术路线 2.0 最终确定为：

```text
Lift / CCR 作为统计主线
+ Markov 作为序列主线
+ GNN 作为结构辅助
+ CrossScore 作为统一融合层
+ 策略层与 Agent 作为落地展示层
```

这是当前数据条件下，最稳、最合理、最适合竞赛与答辩的一条路线。
