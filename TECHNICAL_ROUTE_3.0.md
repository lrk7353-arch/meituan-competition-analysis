# 美团 Cross 项目技术路线 3.0

## 1. 最终方法选型结论

本项目进入 3.0 阶段后的核心技术路线为：

```text
主线方法：Markov + Lift + CCR
新增主线：Task Continuity / Rationality Constraints
统一输出：CrossValueScore + TaskFitScore + PushPriority + Strategy
GNN 状态：弃用（归档，不进入正式主流程）
```

这意味着项目已经不再是：

- “GNN 为主、统计为辅”的建模题
- 或“单一总分榜单”的排序题

而是明确升级为：

> 面向本地生活平台 Cross 导流决策的离线评估框架。

其真正要回答的问题不再只是：

- 哪些 A -> B 会发生
- 哪些 A -> B 转化更强

而是：

- 哪些 A -> B 既有导流价值
- 又不破坏用户当前任务
- 因而值得平台主动推动

---

## 2. 为什么从 2.0 升级到 3.0

技术路线 2.0 已经完成了一个关键转向：

- 放弃 GNN 作为主方法
- 改用 `Lift / CCR + Markov` 作为主干
- 用 `CrossScore` 承接排序与策略层

这个转向本身是正确的。

但继续推进后，出现了新的方法论边界：

### 2.1 2.0 已经解决的问题

- 解决了“单日薄图不适合 GNN 主导”的问题
- 解决了“只看共现、不看成交”的问题
- 解决了“只有统计量、没有统一表达”的问题
- 形成了 `baseline v2 -> CrossScore v1` 的初步闭环

### 2.2 2.0 仍然缺少的问题

2.0 的核心短板不是“模型不够复杂”，而是“目标函数还不完整”。

具体来说：

1. 目前主线更像在回答“哪里最容易承接并成交”
2. 还没有正式回答“这条 Cross 是否合理、是否保任务、是否值得平台主动推”
3. `CrossScore V1` 更像价值排序器，还不是完整的决策评估器

### 2.3 3.0 的核心升级点

因此，3.0 不是加更复杂模型，而是加更高层的约束：

> 从“Cross 潜力评估”升级为“Cross 潜力 × 任务连续性评估”。

也就是说：

- 2.0 更像 `Value Ranking`
- 3.0 才真正变成 `Decision Evaluation`

---

## 3. GNN 的最终状态

在 3.0 中，GNN 不再保留为正式方法组件。

### 明确结论

> GNN 当前处于弃用状态。

原因不是“理论上完全没价值”，而是：

- 当前数据只有单日
- session 逻辑短、图稀疏、POI 粒度细
- 泛化和解释性都不如 `Markov + Lift + CCR`
- 在竞赛答辩和业务表达中，GNN 已不再提供主线增益

### 在项目中的正确处理方式

- 不进入主流程
- 不进入主评分公式
- 不进入主结论叙事
- 若保留，只能作为历史探索或附录失败路线

因此，在 3.0 里：

```text
GNN = archived / deprecated
```

而不是：

```text
GNN = 辅助主线
```

---

## 4. 技术路线 3.0 的整体结构

项目采用“五层主线 + 一输出层”的结构：

```text
数据底座层
-> session 重建 + 订单归因 + 统一口径

候选发现层
-> Markov / Lift / CCR

价值评估层
-> CrossValueScore

合理性约束层
-> TaskFitScore / Rationality Constraints

决策合成层
-> PushPriority / Quadrant / Tier

输出层
-> Strategy / Report / Demo / Agent
```

在职责上：

| 模块 | 角色 | 回答的问题 | 是否主线 |
|---|---|---|---|
| 数据底座层 | 统一行为链路口径 | 数据能否支撑可靠分析 | 是 |
| Markov | 序列承接信号 | 从 A 之后最可能去哪 | 是 |
| Lift | 协同共振信号 | A 与 B 是否存在超预期协同 | 是 |
| CCR | 交易归因信号 | A 出现后是否更容易形成 B 成交 | 是 |
| CrossValueScore | 价值排序层 | 哪些 Cross 值得重点关注 | 是 |
| TaskFitScore | 合理性约束层 | 哪些 Cross 不会破坏主任务 | 是 |
| PushPriority | 最终决策层 | 哪些 Cross 值得平台主动推动 | 是 |
| GNN | 历史探索资产 | 仅用于附录或复盘 | 否 |

## 5. 各方法的职责边界

## 5.1 Markov：序列承接信号

### 作用

回答：

> 给定用户当前处在 A，下一步去 B 的顺路程度有多强。

### 角色理解

- 它描述的是短链路迁移
- 它最接近 session 内即时行为
- 它不是成交价值本身，但决定了“推过去是否顺路”

### 对最终决策的意义

- `Markov` 高，说明适合做即时承接
- `Markov` 低，说明即使 B 很值钱，也未必适合立刻推

---

## 5.2 Lift：协同共振信号

### 作用

回答：

> A 与 B 的协同强度，是否高于 B 的天然热门基线。

### 角色理解

- 它负责从“热门承接”里剥离出“独特协同”
- 它不等于成交，也不等于顺路
- 它最适合解释“为什么这两个场景值得联动”

### 对最终决策的意义

- `Lift` 高，说明有明显协同味道
- `Lift` 低但 `CCR` 高，说明它可能是大盘承接场景，而非独特协同场景

---

## 5.3 CCR：交易归因信号

### 作用

回答：

> A 出现后，同 session 内最终形成 B 订单的概率有多高。

### 角色理解

- 它是最直接的成交代理
- 它让项目从“相关性分析”变成“转化导向评估”
- 它最接近平台干预后的近似收益判断

### 对最终决策的意义

- `CCR` 高，说明这条路更像成交承接链
- `CCR` 低但 `Lift` 高，说明它可能更像内容种草或协同曝光，而非直接转化

---

## 5.4 Task Continuity / Rationality Constraints：合理性约束层

### 作用

回答：

> 即使这条 Cross 有价值，它是否仍然符合用户当前任务，不会造成任务漂移。

这是 3.0 最关键的新增主层。

### 为什么必须加入

因为本地生活不是内容平台。

本地生活推荐必须接受三个约束：

1. 强时空约束
2. 强交易目标
3. 强任务完成导向

因此：

> 好的 Cross 不是“什么都能跳”，而是“有价值且不破坏任务主线”。

### 推荐纳入的合理性信号

1. `R_intent`：主任务保持率  
   A 跳到 B 后，用户是否仍沿着原任务链完成成交。

2. `R_chain`：多跳衰减合理性  
   A -> B 合理，不代表 A -> B -> C -> D 仍合理。

3. `R_semantic`：场景语义距离  
   例如“正餐 -> 奶茶”比“外卖 -> 医美”更近。

4. `R_temporal`：时段匹配度  
   午餐时段适合外卖、奶茶、便利店，不适合医美、家政。

5. `R_terminal`：会话终局质量  
   跳转后是否更容易退出、无单、长时间游离。

### 对最终决策的意义

这一层不负责“发现机会”，而负责：

- 压制荒诞跳转
- 过滤任务漂移
- 把“会发生”转成“该不该推”

---

## 6. 3.0 的双轴评分体系

技术路线 3.0 不再把单一总分作为最终中枢，而是改成双轴体系：

```text
Axis 1: CrossValueScore(A, B)
Axis 2: TaskFitScore(A, B)
```

最终决策建立在这两轴之上。

## 6.1 价值轴：CrossValueScore

建议定义为：

```text
CrossValueScore(A, B)
= α * V_stat
+ β * V_seq
+ γ * V_conv
+ δ * V_cov
+ ε * V_time
```

其中：

- `V_stat`：统计协同价值  
  例如 Lift、PMI、样本平滑

- `V_seq`：序列承接价值  
  例如 Markov 概率、链路位置

- `V_conv`：交易归因价值  
  例如 CCR、conv_sessions

- `V_cov`：覆盖与稳定性  
  例如用户数、session 数、城市/设备稳健性

- `V_time`：时机成熟度  
  例如中位 lag、窗口集中度

它主要回答：

> 这条 Cross 作为业务机会值不值钱。

## 6.2 合理性轴：TaskFitScore

建议定义为：

```text
TaskFitScore(A, B)
= λ * R_intent
+ μ * R_chain
+ ν * R_semantic
+ ξ * R_temporal
+ ο * R_terminal
```

它主要回答：

> 这条 Cross 作为平台动作是否合适。

## 6.3 最终决策分：PushPriority

建议不再简单相加，而采用乘性或门控式融合：

```text
PushPriority(A, B) = CrossValueScore(A, B) * TaskFitScore(A, B)
```

或：

```text
if TaskFitScore < threshold:
    suppress
else:
    rank by CrossValueScore
```

其核心思想是：

> 没有合理性约束的高价值，不等于值得主动推动。

---

## 7. 3.0 的结果表达方式

项目最终不应只给一个榜单，而应输出四象限：

| 象限 | 含义 | 平台动作 |
|---|---|---|
| 高价值 + 高连续性 | 最优 Cross | 强推、资源位联动、重点策略 |
| 高价值 + 低连续性 | 有收益但有漂移风险 | 小流量实验、加强限制与承接 |
| 低价值 + 高连续性 | 合理但收益一般 | 轻触达、弱曝光、补充位 |
| 低价值 + 低连续性 | 不值得推动 | 过滤或仅观察 |

这比单一 `HIGH / MEDIUM / LOW` 更符合平台决策逻辑。

## 8. 业务线与类目的使用边界

3.0 需要明确区分两层结果用途：

### 8.1 业务线层

作为：

- 主报告主图
- 核心答辩结论
- 平台级策略层讨论

因为：

- 结构稳定
- 噪声更低
- 更适合宏观叙事

### 8.2 类目层

作为：

- 案例库
- 精细化策略补充
- 具体玩法展示

因为：

- 更容易受大盘承接场景影响
- 更容易出现长尾高 Lift 幻觉
- 更适合做“示例解释”，不适合直接做主榜

## 9. 当前代码资产在 3.0 中的定位

### 已有资产

- `DATASET_USER_ORDER_REPROFILE.py`  
  负责订单前序归因与统一底座。

- `CROSS_BASELINE_V2.for_colab.py`  
  负责 `Markov / Lift / CCR` 候选与基线资产生成。

- `CROSS_SCORE_V1.py`  
  负责现有价值排序，但仍属于过渡版。

### 3.0 建议新增

- `CROSS_VALUE_V2.py`  
  从 `baseline v2` 资产中提炼更稳的价值轴。

- `TASK_CONTINUITY_V1.py`  
  计算任务连续性与合理性信号。

- `CROSS_DECISION_V3.py`  
  合并 `CrossValueScore + TaskFitScore`，产出四象限与 `PushPriority`。

- `STRATEGY_ENGINE_V3.py`  
  将不同象限与不同流量面映射为运营策略。

### 过渡判断

当前状态可定义为：

```text
baseline v2 = 候选发现层（正式资产）
cross_score v1 = 价值评估层（过渡资产）
technical route 3.0 = 正式主线
```

## 10. 3.0 的最终项目定位

本项目在 3.0 中的最终表述建议统一为：

> 本项目不再尝试重建平台线上推荐模型，而是面向本地生活平台的跨业务导流场景，构建一套“候选发现—价值评估—合理性约束—策略输出”的离线决策评估框架。该框架以 Markov、Lift、CCR 为核心信号，识别高潜 Cross 关系；再通过任务连续性与场景合理性约束，过滤任务漂移风险；最终输出平台真正值得主动推动的 Cross 机会与策略建议。

这就是 3.0 与 2.0 的本质区别：

- 2.0 在回答“哪些 Cross 看起来值钱”
- 3.0 在回答“哪些 Cross 值得平台真正推动”

---

## 11. 3.0 的一句话总结

> 技术路线 3.0 的核心，不是把方法做得更重，而是把目标定义得更准：在本地生活场景里，平台应该主动推动的，不是所有高相关 Cross，而是那些既有导流价值、又不破坏用户任务主线的高质量 Cross。
